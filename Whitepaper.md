## Abstract

Endpoint Detection and Response (EDR) systems are the primary layer of kernel-level behavioural threat detection on modern Windows endpoints. Their instrumentation relies heavily on Event Tracing for Windows (ETW), a kernel facility that exposes memory allocation, thread injection, and system call activity to subscribing consumers. Despite the widespread commercial deployment of EDR products, the defensive community lacks a continuous, reproducible, quantitative model of how code-level mutation affects the volume and diversity of ETW telemetry received by an EDR. 

This paper outlines the design, implementation, and evaluation of the **Secure Telemetry-Driven Code Mutation Framework (STCMF)**, alongside its core engine, **ETWScope**, which directly addresses this gap.

## 1. Introduction

Advanced adversaries proactively evade detection by manipulating their execution patterns. Techniques such as indirect syscalls, API unhooking, call stack spoofing, and polymorphic payload mutation disrupt standard static and dynamic signatures. These variations degrade the richness, frequency, and chronological consistency of the telemetry generated by ETW.

The STCMF integrates three primary subsystems to quantify this degradation:
1. **Code Mutation Engine (CME):** Applies instruction substitution at four discrete intensity levels using Capstone 5.0 and Keystone 0.9.2.
2. **Telemetry Capture and Correlation Engine (TCCE):** Provided by **ETWScope**, a hybrid Rust/Python platform subscribing to ETW providers in real-time.
3. **Test Binaries:** Executables implementing varied syscall invocation paths.

## 2. ETWScope: The Telemetry Capture and Correlation Engine (TCCE)

**ETWScope** acts as the high-performance backbone of the STCMF framework. Designed with a hybrid architecture, it addresses the intense computational requirements of parsing thousands of raw ETW JSON logs while simultaneously applying complex statistical analysis.

### 2.1 Dual Architecture
* **Rust Backend Engine:** Subscribes to the `Microsoft-Windows-Kernel-Process` (KP) and `Microsoft-Windows-Kernel-Audit-API-Calls` (KA) providers natively via Windows APIs, streaming parsed telemetry JSON asynchronously.
* **Python Analysis & TUI:** Provides real-time calculations using a sliding-window approach and displays alerts using a Textual-based Terminal User Interface.

### 2.2 Sigma-Style Detection Engine
ETWScope includes a custom stateful sequence matcher capable of tracking behavioral chains per Process ID (PID). For example, it successfully tracks process injection activity through `EventID(5)` with `TargetPID`, `DesiredAccess`, and `ReturnCode` constraints.

## 3. Mathematical Framework: Telemetry Resilience Score (TRS)

To empirically quantify the disruption of ETW streams between a clean baseline and a mutated payload, ETWScope continuously calculates three primary metrics, converging into the **Telemetry Resilience Score (TRS)**.

### 3.1 Event Volume ($F$) & Shannon Entropy ($H$)
Mutation does not necessarily reduce the raw volume of events; it often increases them due to junk instructions. However, it homogenizes the structural diversity. ETWScope calculates Shannon Entropy over a sliding window:
$$H(X) = - \sum_{i=1}^{n} P(x_i) \log_2 P(x_i)$$

### 3.2 Inter-Event Timing Variance ($CV_t$)
Polymorphic engines disrupt the chronological flow of telemetry. ETWScope measures the Coefficient of Variation for inter-event timing ($CV_t$), highlighting disrupted execution cadence.

### 3.3 TRS Formula
The metrics are normalized and weighted ($w_1=0.45, w_2=0.35, w_3=0.20$) to produce a score between $[0, 1]$:
$$TRS = w_1 V_{ratio} + w_2 E_{ratio} + w_3 T_{cmp}$$

## 4. Empirical Results

Experiments conducted in an isolated two-virtual-machine laboratory running Windows 11 revealed definitive quantitative metrics across four mutation intensity levels ($I=0$ to $I=3$).

**Key Findings:**
1. **Entropy Degradation:** Shannon entropy of the KP provider declined from $1.6476$ bits at baseline ($I=0$) to $1.2356$ bits at $I=2$, a $25.0\%$ reduction.
2. **Timing Variance Explosion:** The mutated payloads caused massive timing disruption ($CV_t$ increasing from $9.78$ to over $173.36$).
3. **TRS Scoring:** The computed TRS ranged from $0.800$ at baseline to $0.783$ at maximum mutation.

### 4.1 Detection Decay Function (DDF)
By fitting TRS across increasing mutation intensities, we modeled the absolute minimum telemetry retention floor ($\epsilon$) using an exponential decay function. The DDF achieved $R^2 \geq 0.90$, with an asymptotic floor of $\epsilon \approx 0.78$. 

This confirms mathematically that **roughly 78% of baseline telemetry persists** regardless of instruction-level mutation, largely because Win32 ETW events are generated deep within kernel routines beyond user-mode reach.

## 5. Conclusion
The STCMF and ETWScope provide a rigorous, mathematical foundation for measuring exactly how much visibility an adversary blinds when deploying specific evasion techniques. By formalizing the Telemetry Resilience Score, security engineering teams can now objectively evaluate endpoint detection blind spots and integrate resilience scoring into continuous integration pipelines.
